# Release Policy

This document describes the release and provenance expectations before Vivi2D
is made public.

## Publication Scope

The repository root stays `private: true`. Workspace packages are publishable
only when `docs/developer/quality/public-api-status.md` marks their publication intent as `experimental`
or `public`.

Initial scope:

- `@vivi2d/web` is the only experimental npm-style package.
- GitHub Releases are the canonical repository-level distribution record for
  source/provenance alpha artifacts.
- Runtime, renderer, provider, native, and C ABI packages remain private until
  their own API, support, security, and legal gates are complete.
- Native and WASM artifacts remain private implementation details unless a later
  release decision explicitly makes them public.

## Support Policy

Vivi2D is pre-1.0. Public or experimental packages may change before a stable
release, but each release must state:

- package version and source commit
- publication intent from `docs/developer/quality/public-api-status.md`
- supported runtime/browser targets
- known limitations for native, WASM, provider, and generated-asset boundaries

Security reporting follows `SECURITY.md`; public issues must not contain
vulnerability details.

## Provenance And Artifacts

Release artifacts must be traceable to the source commit and local quality gate
results.

Quality gate tiers:

- Local minimum for focused changes:
  `git diff --check`, targeted tests, `npm run check:packages-types`, and the
  relevant boundary check for the touched area.
- Pull request required gate: `npm run check:quality` plus any workflow-specific
  checks named in the PR template.
- Release candidate gate: `npm run check:quality:e2e-workflow-record`,
  `gitleaks detect --source . --no-git`, history secret scan, SBOM/notices,
  audits, pack-content checks, and public-release checklist review.
- Public publication gate: release candidate gate plus trusted publishing,
  tarball digest recording, provenance/SBOM attachment, and maintainer
  signoff for public API, security, IP terminology, and license status.

Required before any public npm release:

- `npm run check:quality`
- `npm run build:packages`
- `npm run check:pack-contents`
- `npm run check:release-surface`
- `npm run notices:check`
- `npm run check:sbom`
- `npm run audit:all`
- `npm run audit:prod`

Release artifacts should include:

- a GitHub Release source-review archive and manifest from the reviewed release
  tag for repository-level alpha releases
- package tarballs from the reviewed `dist` outputs
- `THIRD_PARTY_NOTICES`
- a generated CycloneDX JSON SBOM from `npm run sbom:generate`
- provenance generated by npm trusted publishing once npm publication is enabled
- package tarball digests recorded in release notes or the provenance artifact

For an `@vivi2d/web` alpha, npm trusted publishing, SHA-pinned publish
workflows or documented exceptions, SBOM attachment, and tarball digest recording
are release blockers. Native/WASM checksum and signing decisions are blockers
only when native/WASM artifacts become public package artifacts.

For a GitHub Release alpha, follow
`docs/developer/quality/github-release-alpha.md`. The initial asset set is
source/provenance only: source review archive, source review manifest,
repository-wide SBOM, notices, release record, checksums, and release notes. It
must not attach desktop installers, native/WASM binaries, ComfyUI bundles, or
other package artifacts until those surfaces have their own release contract.

Windows desktop installers may be added only after
`docs/developer/quality/windows-installer-alpha.md` is implemented by tracked
workflow checks, an installer release record, checksum verification, signing
status disclosure, and owner approval in the protected release environment.
Installer releases must remain draft pre-releases until a maintainer manually
reviews the asset set and installs the build on a clean Windows machine or VM.

## GitHub And npm Requirements

Future npm publication should use npm trusted publishing and GitHub OIDC
provenance rather than long-lived npm tokens. Publish jobs are the only jobs
that may request `id-token: write`; all other workflows should keep minimal
permissions such as `contents: read`.

Repository settings that must be enabled or explicitly excepted before public
release are tracked in `docs/developer/quality/public-release-checklist.md`.

## Native And WASM Artifact Policy

Public native or WASM binary publication is blocked until:

- native/WASM support policy is written
- checksums are generated for every binary artifact
- signing or attestation mechanism is chosen
- sanitizer/Miri coverage is evaluated where supported
- fuzz or hostile-input coverage is evaluated for exported loaders and ABI entrypoints
- legal/API review approves the ABI and package surface

Until those gates pass, native and WASM outputs are private implementation
artifacts used for conformance and browser smoke testing only.
